# PestManKill UI 模块高分屏适配规划

本文档用于规划 ui 模块的高分屏适配方案，目标是在 Windows 高 DPI 和跨显示器场景下，保证布局稳定、文本清晰、输入准确，并为后续多窗口和多倍率资源扩展提供一致的基础模型。

## 1. 目标

本次高分屏适配要解决以下问题：

- 不同缩放倍率下 UI 布局尺寸不稳定
- 窗口跨屏拖动后文本发糊或命中区域偏移
- 渲染链路、布局链路、输入链路混用逻辑尺寸与物理像素
- 文本缓存没有显式区分设备像素比，导致错误复用低清纹理

最终目标是建立一套统一规则：

- UI 规范坐标统一使用逻辑尺寸
- 物理像素只用于渲染目标、裁剪视口和文本栅格化
- 输入事件在进入 UI 系统前统一转换为逻辑坐标
- 窗口缩放和跨屏变化由统一的运行时窗口度量模型管理

## 2. 现状判断

结合当前 ui 模块实现，可以归纳出以下现状：

- 渲染阶段已经感知窗口像素尺寸
- 窗口平台事件已能监听像素尺寸变化
- 布局、命中测试和状态系统仍默认使用单一尺寸语义
- 文本渲染已经存在逻辑尺寸与栅格尺寸分离的雏形
- 字体和文本缓存目前仍缺少 DPR 感知的缓存隔离

这说明当前问题的根因不是单纯字号偏小，而是窗口逻辑尺寸、窗口像素尺寸和输入坐标三套语义尚未彻底拆开。

## 3. 设计原则

### 3.1 统一逻辑坐标空间

所有 UI 组件、布局结果、命中测试、滚动偏移、文本字号统一使用逻辑单位。逻辑坐标空间是 UI 系统唯一的规范空间。

### 3.2 像素尺寸只在渲染阶段生效

窗口像素尺寸用于：

- GPU 渲染目标尺寸
- Scissor 和 Viewport
- 文本和图标的实际栅格尺寸

像素尺寸不应直接回写组件的 Size，也不应污染 Yoga 布局根节点尺寸。

### 3.3 输入统一归一化

所有来自 SDL 的原始鼠标和触控输入，应在进入 UI 事件队列前统一映射到逻辑坐标。后续 HitTest 和 StateSystem 只消费逻辑坐标。

### 3.4 运行时统一管理窗口度量

窗口逻辑尺寸、像素尺寸、设备像素比、显示器缩放倍率等信息应集中收敛到统一的 Window Metrics 模型，而不是分散在多个系统内通过隐式约定传递。

## 4. 核心方案

### 4.1 建立 Window Metrics 运行时模型

建议为每个窗口维护统一的运行时度量结构，至少包含：

- windowID
- logicalSize
- pixelSize
- devicePixelRatio
- contentScale
- metricsDirty

这套模型建议挂接在现有 UI 全局上下文或运行时上下文中，作为布局、渲染、输入三条链路的统一数据来源。

### 4.2 固定组件层的尺寸语义

需要重新明确组件语义边界：

- Size 表示逻辑尺寸
- Position 表示逻辑位置
- CanvasSize 表示逻辑画布尺寸
- Text.fontSize 表示逻辑字号

一旦语义固定，平台层和渲染层就不能再把像素值直接写回这些组件。

### 4.3 收敛窗口事件同步入口

窗口创建、尺寸变化、像素尺寸变化、窗口跨屏移动等事件，需要统一进入一个 refreshWindowMetrics 入口。该入口负责：

- 读取窗口逻辑尺寸
- 读取窗口物理像素尺寸
- 读取显示器缩放倍率
- 计算当前 devicePixelRatio
- 判断是否需要重建文本纹理或刷新相关缓存

这样可以避免多个 system 各自读取 SDL 状态并形成不一致行为。

### 4.4 修正根布局尺寸来源

当前高分屏适配里最关键的一点，是根布局尺寸必须使用逻辑尺寸而不是像素尺寸。

如果把窗口像素宽高直接写回根节点 Size，会带来以下连锁问题：

- Yoga 计算结果失真
- 换行宽度错误
- 滚动区域尺寸异常
- 命中区域与视觉位置不一致

因此窗口像素变化只能更新 Window Metrics，不能直接更新布局组件。

### 4.5 统一输入坐标转换

建议在 RawPointer 事件入队前，统一完成 pixel 到 logical 的坐标转换。

可采用以下策略：

- 保留原始像素坐标用于调试
- 同时存储逻辑坐标作为业务处理输入
- 后续交互、命中测试、拖拽、滚动系统只消费逻辑坐标

这样可以隔离平台差异，减少高 DPI 下鼠标偏移问题。

### 4.6 扩展 RenderContext

渲染上下文需要显式拆分两套尺寸：

- logicalScreenWidth / logicalScreenHeight
- pixelScreenWidth / pixelScreenHeight
- devicePixelRatio

布局生成的顶点、裁剪矩形和绘制区域先保持逻辑单位，真正提交到 GPU 前再映射为像素空间。这样渲染系统可以明确承担坐标映射职责，而不是把像素语义提前泄漏到布局侧。

### 4.7 文本渲染与缓存改为 DPR 感知模型

文本系统建议采用 逻辑字号 + 栅格倍率 的模型：

- 逻辑字号继续作为 UI 设计层输入
- 实际栅格字号按 oversampleScale 与 devicePixelRatio 共同计算
- 字体缓存和文本纹理缓存必须把 rasterScale 或等价 DPR 信息纳入缓存 key

否则在跨屏拖动或切换缩放倍率后，系统会继续复用旧贴图，导致文字变糊。

### 4.8 图片和图标资源分阶段处理

文本链路应优先完成高分屏适配，因为收益最大且最影响可读性。对于 Image 和 Icon 建议分阶段推进：

- 第一阶段允许位图资源继续通过运行时缩放显示
- 第二阶段再评估是否引入 1x / 2x 资源版本
- 若已有矢量图标路径，则优先走矢量渲染

这样可以保证首版方案先解决核心问题，不被资源生产流程阻塞。

## 5. 分阶段实施计划

### 阶段一：建立 DPI 运行时基础设施

任务目标：

- 新增统一的 Window Metrics 结构
- 在窗口创建后完成首次 metrics 初始化
- 为每个窗口缓存 logicalSize、pixelSize 和 devicePixelRatio

交付结果：

- 高 DPI 相关状态从零散逻辑变为统一上下文
- 后续系统可以从统一入口读取窗口度量信息

### 阶段二：拆分布局尺寸与像素尺寸

任务目标：

- 固定 Size、Position、CanvasSize 等组件语义
- 移除窗口像素尺寸直接覆盖根节点 Size 的逻辑
- 让 LayoutSystem 根节点始终使用逻辑尺寸

交付结果：

- 高 DPI 不再直接影响 Yoga 根布局的语义
- 布局、滚动、换行逻辑稳定下来

### 阶段三：统一输入归一化

任务目标：

- 在事件层完成输入坐标从 pixel 到 logical 的转换
- 命中测试和交互系统全部只消费逻辑坐标

交付结果：

- 鼠标悬停、点击、拖拽、滚动的命中区域与视觉位置一致

### 阶段四：渲染上下文双尺寸化

任务目标：

- 扩展 RenderContext，使其同时具备逻辑尺寸与像素尺寸
- GPU 提交前统一完成坐标映射和裁剪映射

交付结果：

- 渲染职责边界清晰
- 渲染与布局之间的尺寸语义彻底解耦

### 阶段五：文本清晰度与缓存隔离

任务目标：

- 改造 TextRenderer、FontManager、TextTextureCache
- 建立 DPR 感知的文本栅格与缓存策略

交付结果：

- 不同缩放倍率和跨屏场景下文本保持清晰
- 不再错误复用低倍率纹理

### 阶段六：验证与回归

任务目标：

- 补充单元测试和手工验证清单
- 覆盖常见缩放倍率与跨显示器场景

交付结果：

- 高分屏改造具备持续回归能力

## 6. 建议改造范围

建议重点围绕以下模块推进：

- GlobalContext 或运行时上下文入口
- Factory 窗口创建路径
- PlatformWindowSystem
- StateSystem
- LayoutSystem
- InteractionSystem
- HitTestSystem
- RenderContext
- RenderSystem
- FontManager
- TextRenderer
- TextTextureCache

这些模块分别承担窗口度量、输入归一化、布局语义、渲染映射和文本缓存隔离，是高分屏适配的主路径。

## 7. 验证方案

### 7.1 自动化验证

建议新增以下测试：

- Window Metrics 计算是否正确
- pixel 与 logical 坐标转换是否正确
- 窗口像素尺寸变化不会污染根布局尺寸
- 文本缓存能够按不同 rasterScale 正确隔离

### 7.2 手工验证

至少覆盖以下场景：

- 100% 缩放
- 150% 缩放
- 200% 缩放
- 窗口从低 DPI 显示器拖到高 DPI 显示器
- 窗口从高 DPI 显示器拖回低 DPI 显示器

重点检查内容：

- 按钮、边框、圆角和文本尺寸是否一致
- 文本是否清晰且不会突然发糊
- 鼠标悬停和点击位置是否准确
- 滚动条拖拽是否与视觉位置一致
- Slider、TextEdit 光标、文本选择区域是否偏移

## 8. 风险与边界

需要明确以下风险点：

- 若继续允许像素尺寸进入组件层，后续所有 system 都会继续耦合
- 若输入不先做归一化，命中测试问题会在多个控件上重复出现
- 若缓存 key 不带 DPR，文本清晰度问题会持续存在

本规划当前不强制解决的内容：

- 全量位图资源多倍图生产体系
- UI 之外的 client 或 server 逻辑调整
- 资源管线层面的历史图片资产重制

## 9. 推荐落地顺序

推荐按以下顺序实施：

1. 先建立 Window Metrics 运行时模型
2. 再修正布局根尺寸语义
3. 然后统一输入坐标转换
4. 接着扩展 RenderContext
5. 最后处理文本缓存与回归测试

该顺序的优势在于先解决坐标语义，再处理渲染清晰度，能避免后续反复返工。

---

结论：

ui 模块的高分屏适配，本质上是一次坐标语义和窗口度量模型的统一，而不是单纯调整字号或放大窗口。只要先把逻辑尺寸、物理像素和输入坐标三条语义彻底拆开，后续布局稳定性、文本清晰度和交互准确性都会自然落到正确的位置。
