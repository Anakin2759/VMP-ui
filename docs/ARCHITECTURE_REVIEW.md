# UI 架构锐评与边界

本文档是"直话直说版"评审：优点保留，问题点名，优先级明确。

## 先说结论

- 这是一个工程化程度不低的 UI ECS 框架：模块齐全、渲染分层清晰、测试在补齐。
- 但核心矛盾仍然是"全局单例便利性"与"可测试/可组合性"的冲突。
- 如果继续放任跨系统隐式耦合，后续多窗口场景会越来越难维护。

## 目前做得好的地方

1. 渲染职责切分正确
- `RenderSystem` 作为协调层，配合 `IRenderer`、`BatchManager`、`CommandBuffer` 分工明确。
- 这条线可扩展性较好，新增渲染类型成本可控。

2. 运行时上下文开始收敛
- `UiRuntime` 负责持有运行时实例，`UiRuntimeScope` 负责激活作用域，新增和迁移代码已开始通过 `RuntimeFacade` 读取 registry、dispatcher 和上下文。
- `Registry`/`Dispatcher` 静态接口仍在，但当前更适合视为兼容层，而不是新的边界中心。

3. 常规帧路径已经收口到单点
- `Application` 主循环单点消费 `QueuedTask -> InputTask -> RenderTask`。
- `TaskChain` 仍然存在，但当前职责是主循环内部实现，不再单独承担公共入口语义。

4. 工厂和 DSL 提升了 UI 构建效率
- `factory` + `chains` 语法在业务层可读性高，适合快速搭建界面。

## 核心问题（按严重度）

1. 全局访问太强，边界太弱
- 虽然 `RuntimeFacade` 已成为受控入口，但 `Registry`/`Dispatcher` 静态调用仍在大量旧路径中共存。
- 结果：调用方便，但依赖隐藏，回归问题定位困难。

2. 系统之间存在"事件驱动 + 共享状态"双耦合
- 例如交互、命中、状态更新并行依赖事件流和全局组件。
- 结果：时序问题容易变成"偶现"，并且难写稳定单测。

3. 输入语义与渲染语义尚未完全解耦
- 原始输入、逻辑坐标、像素坐标在不同系统中仍有交叉使用。
- 结果：跨窗口、滚动容器场景容易出现命中偏差和行为分歧。

4. 工厂默认策略历史包袱较重
- 工厂函数默认值会直接影响布局行为，若策略不统一会产生测试脆弱性。
- 结果：看似"小改动"，实际会击穿集成测试预期。

## 架构边界建议（务实版）

1. 保留全局入口，但把"写操作"逐步收敛到 Runtime Facade
- 先不追求一次性去单例，避免大爆炸重构。
- 原则：系统读可放宽，写优先走受控入口。

2. 把输入链路做成单向语义转换
- SDL 原始输入 -> 逻辑输入事件 -> HitTest/State。
- 任何像素语义只在平台层和渲染提交前出现。

3. 约束系统职责
- InteractionSystem 只做采集与分发。
- HitTestSystem 只负责空间判断。
- StateSystem 当前是状态写入和窗口状态同步的唯一注册入口，同时承担 Hover/Active/Focus、滚动条/滑块交互和帧尾状态刷写；它还没有真正瘦到只剩纯状态机。
- RenderSystem 只负责可视化提交。

4. 文档与测试一起演进
- 每次改系统边界，必须补至少一条跨系统集成测试用例。

## 近期优先事项

1. 为输入链路补充时序测试与多窗口测试。
2. 继续削弱跨系统静态调用深度，优先把新增和迁移代码收口到 `RuntimeFacade`。
